# CLAUDE.md — Модуль получения данных (Фаза 0, v1: выгрузка Контур.Закупок в Excel)

Положи в корень репозитория сервиса (или папку `/ingestion`).
Это инструкция для тебя, Claude Code: что строим, как и в каком порядке.
Актуальный бриф; заменяет предыдущие варианты (ЕИС-SOAP, DaMIA-only, парсинг писем).

---

## 1. Цель

Модуль получения данных о закупках со СМЕННЫМ источником. На v1 источник —
**Excel-выгрузка из Контур.Закупок** (пользователь нажимает «Выгрузить в Excel» и
кладёт файл). Парсим, нормализуем, складываем в PostgreSQL. Это первый кусок
системы анализа тендеров для архитектурного бюро.

Поток v1:
```
Контур → «Выгрузить в Excel»  ->  .xlsx  ->  ExcelSource (парсинг)  ->  нормализация  ->  Postgres  ->  очередь на анализ
```

Никакого API, пересылки писем и скрейпинга сайта на v1 не нужно. Выгрузка идемпотентна:
повторная загрузка обновляет существующие и добавляет новые (upsert по номеру).

Архитектурная идея: источник — это адаптер за общим интерфейсом, чтобы позже
подключить почту/DaMIA/ЕИС, не переписывая остальное.

## 3. Формат файла (по реальной выгрузке — сверь с фикстурой)

Реальный образец положи в `tests/fixtures/` (обезличивать не нужно — данные публичные).
Что известно по образцу `Контур_Закупки_*.xlsx`:

- Один лист (имя = название организации, напр. «Крафт Групп»). Бери активный/первый лист,
  имя НЕ хардкодь.
- **Заголовок в ДВУХ строках:** строка 1 — групповые шапки (Закупка / Заказчик / Результат),
  строка 2 — реальные названия колонок. Читать через `header=1` (pandas) — то есть данные
  с 3-й строки.
- 34 колонки. Порядок и имена (строка 2):
  `Номер, Название, НМЦ, Обеспечение заявки, Обеспечение контракта,
  Обеспечение по поставке товара или выполнению работы,
  Обеспечение по обслуживанию, эксплуатации, ремонту и (или) утилизации,
  Обеспечение гарантийных обязательств, Аванс, Валюта закупки, Дата публикации,
  Планируемая дата публикации, Окончание приема заявок, Проведение отбора, Этап отбора,
  Тип торгов, Ссылка на ЕИС, Способ отбора, ЭТП, СМП, СОНО, Метка, Комментарий,
  Ответственный, Регион(заказчик), Название(заказчик), ИНН, КПП, Место поставки,
  Место поставки из документов, Публикация протокола, Название победителя,
  ИНН победителя, КПП победителя, Предложение победителя`.
  Колонки заказчика «Регион/Название/ИНН/КПП» в pandas получат суффикс `.1` — учти.

### Подводные камни (наблюдаемые в реальных данных)
- **Номер — это СТРОКА, не число.** Бывает 19 цифр (44-ФЗ: `0373200597926000045`),
  ~11 цифр (223-ФЗ: `32616156693`), короткий ID (`209695`), и буквенно-цифровой для
  коммерческих (`B2606261843476`, `ГП642268`, `SBR028-2606260048`). НИКОГДА не приводить к int.
- **Даты — Excel-serial** (число вроде `46203.48`). Конвертация: база `1899-12-30`,
  единица — дни. ОБЯЗАТЕЛЬНО guard по диапазону (напр. 30000–60000), иначе мусорные
  значения дают переполнение. Не прошло проверку -> null.
- **Обеспечения и Аванс — смешанный тип:** в одних строках абсолютная сумма в рублях
  (`164799.27`), в других — процент строкой (`30.00 %`, `5.00 %`), в Авансе бывает
  `Есть. См. документацию`. Сохраняй СЫРОЕ значение как есть + распарсенное число и флаг
  (рубли/процент), где удалось.
- **Тип торгов:** значения `44-ФЗ`, `223-ФЗ`, `Коммерческие`, `615 ПП`.
- **Регион** в виде `77 Москва` — разбей на `region_code` (`77`) и `region_name` (`Москва`).
- **Ссылка на ЕИС** — это гиперссылка в ячейке (видимый текст «ЕИС»). URL при желании
  достаётся через `openpyxl` `cell.hyperlink.target`, но номер и так есть — ссылка не нужна.
- **Дубли по объекту:** одна стройка может быть несколькими строками с разными номерами и
  ЭТП (одна закупка размещена на разных площадках). По номеру это РАЗНЫЕ записи — это норма,
  не схлопывай на этом уровне (это задача фильтра/карточки позже). Upsert идёт по номеру.

## 4. Архитектура источника (адаптеры)

```
SourceAdapter (ABC)
  def fetch() -> Iterable[RawTender]

ExcelSource(SourceAdapter)     # v1 — парсинг выгрузки Контура (путь к .xlsx)
# EmailSource / DamiaSource / EisSource — позже, тот же интерфейс
```
`RawTender` — общий нормализованный объект (pydantic): `reestr_number`, `source`, поля
закупки/заказчика, `raw` (исходные значения строки dict). Pipeline работает только с
`RawTender` и ничего не знает про конкретный источник.

## 5. Стек

- **Python 3.12**.
- Чтение Excel: `openpyxl` (для гиперссылок) и/или `pandas` (`read_excel`, `header=1`).
- Модели/валидация: `pydantic`.
- БД: **PostgreSQL 16** (без Supabase), `psycopg` (v3) или `SQLAlchemy`, миграции `alembic`.
- Логи: `structlog`/`logging` в JSON. Контейнеры: `Dockerfile` + общий `docker-compose.yml`
  (`postgres` + при необходимости `worker`).
- Зависимости: `uv` или `pip` + `pyproject.toml`. Качество: `ruff`, `mypy`, `pytest`.

## 8. Задачи по порядку (коммить по шагам)

1. Скелет проекта: зависимости, `ruff`/`mypy`/`pytest`, `.env.example`.
2. Положить реальную выгрузку в `tests/fixtures/`.
3. БД + миграции: Postgres в Docker, схема (раздел 9), первая миграция alembic.
4. `sources/base.py`: `SourceAdapter` (ABC) + `RawTender` (pydantic).
5. `sources/excel_source.py`: чтение листа (активный, `header=1`), маппинг 34 колонок,
   сохранение сырой строки в `raw`.
6. `normalize.py`: даты (Excel-serial + guard), обеспечения/аванс (₽/% + сырьё),
   регион (код+имя), номер как строка. Покрыть `pytest` на фикстуре.
7. `pipeline.py` + idempotent upsert по `reestr_number` (ON CONFLICT DO UPDATE) + enqueue.
8. `cli.py`: `ingest --file <path>` (загрузка одной выгрузки).
9. README: как поднять, как загрузить выгрузку, как читать журнал.

## 9. Схема БД (минимум для Фазы 0)

- `tenders` — нормализованная закупка:
  `reestr_number` (TEXT, уникальный), `source`, `subject` (Название), `nmck` (numeric),
  `currency`, `law` (Тип торгов: 44/223/комм/615), `purchase_method` (Способ отбора),
  `stage` (Этап отбора), `etp`, `smp_sono` (bool/текст), `publish_date`,
  `submission_deadline`, `customer_name`, `customer_inn`, `customer_kpp`,
  `region_code`, `region_name`, `delivery_place`,
  `advance_raw`, `advance_pct` (nullable), `created_at`, `updated_at`.
- `tender_securities` (или JSONB-поле в tenders) — 5 видов обеспечения:
  по каждому `raw_value` + `amount_rub` (nullable) + `percent` (nullable).
- `tender_result` (nullable) — `protocol_date`, `winner_name`, `winner_inn`, `winner_offer`.
- `tender_raw` — сырьё: `reestr_number`, `source`, `payload` (JSONB всей строки), `fetched_at`.
- `analysis_queue` — `reestr_number`, `status`, `enqueued_at`.
- `ingestion_runs` — журнал: `started_at`, `finished_at`, `source`, `file`,
  `rows_total`, `tenders_upserted`, `parse_failures`, `status`, `error`.

Upsert по `reestr_number` — повторная выгрузка не плодит дубли, обновляет изменившееся.

## 11. Критерии готовности (Definition of Done)

- `docker compose up` поднимает Postgres.
- `cli ingest --file tests/fixtures/<выгрузка>.xlsx` парсит лист (≈160 строк в образце),
  нормализует, кладёт в `tenders` (+ сырьё в `tender_raw`), ставит в `analysis_queue`;
  повторный прогон того же файла не плодит дубли (upsert по номеру).
- Даты, обеспечения (₽/%), регион (код+имя) корректно разобраны; номер хранится строкой.
- Тесты на фикстуре зелёные; `ruff` и `mypy` чистые.
- Pipeline работает через `SourceAdapter` (не завязан на Excel жёстко).
