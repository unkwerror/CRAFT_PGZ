# tender-ingest (Фазы 0–1)

Система анализа закупок для архитектурного бюро. Реализовано:

- **Фаза 0 — приём данных.** Источник — Excel-выгрузка из Контур.Закупок
  (пользователь жмёт «Выгрузить в Excel»). Парсим, нормализуем, кладём в PostgreSQL.
- **Фаза 1 — скоринг.** Многофакторная оценка каждой закупки под профиль **КРАФТ ГРУПП**
  (craft72.ru): score 0–100 + краткое резюме от Claude по всем полям карточки.

```
Контур → .xlsx → ExcelSource → нормализация → Postgres → analysis_queue
                                                              ↓
                       факторы (закон/способ/аванс/цена/регион/стоп-лист)
                          ├─ жёсткое исключение (аукцион/Россети/СНГ/этап) → noise
                          └─ иначе Claude (полная карточка, пачками) → score + резюме
```

Границы фаз — в [`CLAUDE.md`](./CLAUDE.md). Архитектура (адаптеры, скоринг,
рубрика) — в [`docs/architecture.md`](./docs/architecture.md).

## Архитектура

- `sources/` — `SourceAdapter.fetch() -> Iterable[RawTender]`; v1 — `ExcelSource`.
  Pipeline работает только с `RawTender`, про Excel не знает.
- `relevance/factors.py` — объективные факторы из полей (закон, конкурс/аукцион,
  аванс, ценовой диапазон 2–150 млн, приоритетные регионы, стоп-лист заказчиков) +
  жёсткие исключения (дёшево, без LLM). **Аналитик правит регионы/диапазон здесь.**
- `relevance/arbiter/` — `RelevanceArbiter.decide_batch(...)`; `ClaudeArbiter`
  (Anthropic API): рубрика скоринга и профиль КРАФТ в `prompt.py`, structured output,
  батч-запросы. Нужен `ANTHROPIC_API_KEY`.
- `relevance/scorer.py` — оркестрация: факторы → (не исключено) Claude пачками → score.
- `normalize.py`, `db/` — нормализация полей; модели/репозитории/очередь/журнал.

## Использование

```bash
# 1) загрузить выгрузку
tender ingest --file "tests/fixtures/Контур.Закупки_30.06.2026.xlsx"
# → rows=160 upserted=160 failures=0 status=success

# 2) оценить всё, что в очереди (score 0–100 + резюме)
tender score
# → total=160 relevant=38 maybe=45 noise=77 claude_calls=120
```

Обе команды **идемпотентны**: повторная загрузка того же файла не плодит дубли
(upsert по `reestr_number`); `score` берёт только закупки со статусом `pending` в
очереди и переводит их в `scored`.

### Скоринг (Claude)

Оценку даёт Claude по всей карточке; ключ обязателен:

```env
ANTHROPIC_API_KEY=sk-ant-...
CLAUDE_MODEL=claude-opus-4-8   # дешевле: claude-haiku-4-5
```

- **Факторы** (`factors.py`) считаются из полей без LLM; жёсткие дисквалификаторы
  (электронный аукцион, заказчики Россети/ЕЭСК, закупки СНГ, неактивный этап) →
  сразу `noise`, экономя вызовы.
- **Claude** оценивает остальное по рубрике (профиль КРАФТ, закон, конкурс, аванс,
  цена 2–150 млн, приоритетные регионы УрФО+Пермь+Омск, исключение дорожных работ) —
  даёт score 0–100 и резюме. Вердикт: ≥60 relevant, ≥35 maybe, иначе noise.
- Карточки шлются **пачками** (≈20 в запросе, батчи параллельно) — быстро и дёшево;
  пропущенные моделью строки ретраятся; сбой не роняет прогон.

## Что лежит в БД

| Таблица | Назначение |
|---|---|
| `tenders` | нормализованная закупка (PK `reestr_number`); обеспечения и результат — JSONB |
| `tender_raw` | сырьё строки целиком (JSONB) — ничего не теряем при смене формата |
| `analysis_queue` | очередь: `pending` → `scored` |
| `tender_relevance` | `score`, `verdict`, `decided_by` (rules/claude), `summary`, `factors` |
| `ingestion_runs` | журнал прогонов: файл, строк, upsert, ошибок, статус |

Прозрачность: по каждой закупке видно `score`, резюме (`summary`), объективные
факторы (`factors`) и кто принял решение (`decided_by`).

## Логи

`structlog` в JSON: `ingest_done` со счётчиками, `score_done` с распределением
вердиктов, `tender_upsert_failed` по плохим строкам (строка не валит весь файл —
SAVEPOINT на строку).
