## Приоритет 1 — сбой батча не должен хоронить целую пачку тендеров

**Файл:** `src/tender_ingest/relevance/arbiter/claude.py`, `decide_batch`

Если запрос к Claude падает целиком (сеть/лимит/невалидный JSON) — `by_reestr`
остаётся пустым, и КАЖДАЯ карточка из батча (до `CHUNK_SIZE=20`) получает
`fallback = ArbiterVerdict(score=0, summary=NOT_SCORED, ...)`. `score=0` →
`verdict_from_score` → `"noise"`. После одного неудачного ретрая (`scorer.py`,
блок `missed`) партия из ~20 реальных тендеров молча ложится в шум. Отличить их
от «Claude реально посчитал, что не подходит» можно только читая `summary` вручную.

**Правка:** fallback-score должен приземляться в зону `maybe`, а не `noise` —
это гарантирует, что неоценённые карточки попадут на глаза человеку, а не
потеряются. Конкретно:

```python
# arbiter/claude.py
fallback = ArbiterVerdict(score=NOT_SCORED_SCORE, summary=NOT_SCORED, provider=self.provider)
```
где `NOT_SCORED_SCORE` — константа со значением строго между `MAYBE_THRESHOLD` и
`RELEVANT_THRESHOLD` (например 45) — попадает в `maybe`, никогда не в `noise` и
не в `relevant` автоматически. Вынести это значение в `base.py` рядом с
`NOT_SCORED`, чтобы `scorer.py` не хардкодил магическое число.

Не менять сам механизм ретрая — он адекватен (один повторный заход на пропущенные).
Меняется только то, куда падает результат, если и повтор не помог.

## Приоритет 2 — модель по умолчанию

**Файлы:** `src/tender_ingest/config.py`, `.env.example`

`claude_model: str = "claude-opus-4-8"` → `"claude-haiku-4-5"`. Задача — взвешенная
оценка по структурированным полям + короткое резюме, батчами; это по силам Haiku
и кратно дешевле при объёме в десятки карточек за прогон. Перед тем как оставить
Haiku на постоянку — прогони 15–20 карточек на Haiku и Sonnet параллельно, сравни
score/резюме на пограничных случаях (смешанные обеспечения, ЛЭП vs дороги); если
разницы в качестве не видно — Haiku остаётся дефолтом.

## Приоритет 3 — без ключа Claude сейчас не оценится НИЧЕГО (не только LLM-часть)

**Файлы:** `src/tender_ingest/relevance/arbiter/factory.py`,
`src/tender_ingest/relevance/scorer.py`

`create_arbiter()` бросает `ValueError` без ключа; `score_pending` вызывает его
ОДИН раз в самом начале, до того как посчитаны `factors`/`hard_exclusion` хоть для
одной закупки. Раньше (до батчинга) без ключа хотя бы правила отрабатывали — сейчас
без ключа `score` не обработает вообще ни одной строки, хотя жёсткие исключения
(аукцион/стоп-лист/этап) в ключе не нуждаются.

**Правка:** в `score_pending` — считать `factors`/`hard_exclusion` для всех
закупок ДО обращения к арбитру; арбитра создавать лениво и только если
`llm_jobs` не пуст; если создание падает — не пробрасывать исключение, залогировать
`arbiter_unavailable` с числом пропущенных, оставить эти закупки в очереди со
статусом `pending` (не писать им score вовсе). Итог: жёсткие исключения обработаны
всегда, LLM-часть — best-effort.

## Приоритет 4 — проверка СНГ смотрит не туда

**Файл:** `src/tender_ingest/relevance/factors.py`, `is_sng`

`is_sng = "снг" in law_l`, где `law_l` — поле «Тип торгов». В реальной выгрузке
Контура это поле принимает только 44-ФЗ/223-ФЗ/Коммерческие/615 ПП — слова «снг»
там не бывает (проверено на фикстуре). Тест `test_sng_and_inactive_stage`
проверяет придуманное значение поля, а не то, что реально приходит, и поэтому
зелёный, ничего не гарантируя на живых данных.

**Правка:** нужен реальный признак закупок СНГ/зарубежа в выгрузке — скорее
всего это регион/место поставки (страна вместо кода субъекта РФ) или отдельная
колонка, которой сейчас нет в модели. Пока такого признака нет — либо убрать
проверку как не работающую, либо (лучше) отдать это на семантику Claude через
`SYSTEM_PROMPT` (добавить явную инструкцию: «если регион/место поставки — не
регион РФ, а другая страна — это жёсткое исключение, score=0»), раз в `factors.py`
надёжно определить это сейчас нечем.

## Приоритет 5 — стоп-лист заказчиков и активный этап: точность матчинга

**Файл:** `src/tender_ingest/relevance/factors.py`

- `customer_excluded` матчится regex по имени (`россети|еэск|...`). Работает для
  тестовых примеров, но чувствителен к вариантам написания. Если/когда появится
  таблица известных ИНН (обсуждали ранее) — матчить по ИНН как основной признак,
  имя оставить как быстрый эвристический пре-фильтр.
- `stage_active` — точное сравнение с строкой `"подача заявок"`. Стоит явно
  сверить с полным списком реальных значений колонки «Этап отбора» в выгрузках
  (не только в фикстуре) — если там встречаются варианты формулировки, точное
  сравнение будет молча считать их неактивными.
