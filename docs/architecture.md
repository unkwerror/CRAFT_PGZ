# Архитектурные решения (ADR, кратко)

## ADR-001 — Источник как сменный адаптер (`SourceAdapter`)

**Контекст.** Источник данных о закупках со временем меняется: на v1 это
ручная Excel-выгрузка из Контур.Закупок, позже — почта (пересылка писем), DaMIA
(API), сервисы отдачи ЕИС. Бизнес-логике (нормализация, upsert, очередь) всё
равно, откуда пришли данные.

**Решение.** Общий интерфейс `SourceAdapter.fetch() -> Iterable[RawTender]`.
`RawTender` — нормализованный объект (pydantic) с полями закупки/заказчика и сырой
строкой (`raw`). Pipeline работает только с `RawTender` и ничего не знает про
конкретный источник.

- **v1 (сейчас):** `ExcelSource` — парсинг выгрузки Контура (`.xlsx`).
- **Позже:** `EmailSource` / `DamiaSource` / `EisSource` — тот же интерфейс,
  без переписывания pipeline/repository.

Преждевременной абстракции не плодим: пока есть один адаптер, но интерфейс
зафиксирован, чтобы добавление источника не ломало остальное.

## ADR-002 — Идемпотентный upsert по `reestr_number`

Повторная загрузка той же выгрузки не плодит дубли: ON CONFLICT (`reestr_number`)
DO UPDATE для `tenders` и `tender_raw`. Очередь анализа пополняется ON CONFLICT DO
NOTHING — повторный прогон не сбрасывает статус уже взятых в работу закупок.

«Дубли по объекту» (одна стройка размещена на нескольких ЭТП разными номерами) на
уровне источника **не схлопываем** — по номеру это разные записи. Дедуп по объекту —
задача фильтра/карточки следующей фазы.

## ADR-003 — Сырьё строки храним целиком (`tender_raw.payload`, JSONB)

Нормализуем только известные поля; всю исходную строку (34 колонки) сохраняем в
JSONB. Если формат выгрузки изменится или понадобится перепарсить — данные не
потеряны. Парсер валидирует структуру (имена/число колонок) и падает явной ошибкой
`ExcelFormatError`, если выгрузка не та.

## ADR-004 — Устойчивость к грязным данным

Нормализаторы — чистые функции с guard'ами (см. `normalize.py`):
- **номер** — всегда строка (ведущие нули, буквенные ID коммерческих закупок);
- **даты** — `datetime` от openpyxl как есть; число трактуем как Excel-serial с
  guard по диапазону (мусор → null);
- **обеспечения/аванс** — смешанный тип: рубли, процент строкой (`30.00 %`) или
  свободный текст (`Есть. См. документацию`) → сохраняем сырьё + распарсенное;
- **регион** — `77 Москва` → код + имя.

Одна плохая строка не валит весь файл: upsert каждой строки в отдельном SAVEPOINT,
сбой считается в `parse_failures` журнала прогона.

## ADR-005 — Релевантность: правила + LLM-арбитр (Фаза 1)

**Контекст.** Выгрузка Контура **уже отфильтрована** под профиль бюро (в Контуре
настроен поиск): из 160 строк ~90% — проектные/изыскательские/реставрация/
благоустройство. В выгрузке **нет ОКПД2**, поэтому фильтруем по тексту предмета
(колонка «Название»). Задача фильтра — не грубо резать, а **ранжировать** и убирать
явный шум.

**Решение.** Гибрид:
1. **Правила** (`relevance/rules.py`) — быстрый прозрачный скоринг по ключевым
   словам профиля (+вес) и анти-словам (−вес). Решают ясные случаи.
2. **LLM-арбитр** (`relevance/arbiter/`) — только для спорных (maybe). Реализация
   `ClaudeArbiter` (Anthropic API) за интерфейсом `RelevanceArbiter`. Экономит
   вызовы: LLM зовётся лишь на ~единицы закупок из выгрузки.

Почему правила, а не сразу эмбеддинги/pgvector: объёмы крошечные (160 строк), нужна
**прозрачность** (видно, *почему* релевантно), минимум LLM-вызовов. Семантику
добавим позже, если правила начнут пропускать нестандартные формулировки.

**Recall-first: сильный сигнал доминирует над анти.** Анти-слова (поставка/аренда/
ГСМ/медоборудование) часто описывают **объект**, который бюро проектирует (склад
ГСМ, медцентр), а не суть закупки. Поэтому при сильном проектном сигнале (≥30) и
`pos > anti` вердикт `relevant`; при `pos == 0` — `noise`; иначе `maybe` → арбитру.
Так «проектная документация склада ГСМ» не теряется. Цена — редкий ложноположительный
(аналитик пролистнёт), что дешевле пропущенного реального тендера.

**Append-friendly и идемпотентно.** Результат — в `tender_relevance` (upsert по
номеру), очередь двигается `pending` → `scored`. Повторный `score` не пере-оценивает
уже обработанное. Полный аудит-след: `score`, `matched`/`anti_matched`, `decided_by`,
`llm_reason`.
