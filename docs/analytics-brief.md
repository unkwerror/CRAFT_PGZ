# Бриф: модуль аналитики тендеров + ИИ-экономист (RAG)

Дата: 2026-07-03. Статус: принят в работу (фазы A–C реализуются).

## 1. Контекст и пробел в данных

В базе есть всё для аналитики **по рынку**: `publish_date`, `submission_deadline`,
`nmck`, регион, закон, способ закупки, заказчик (ИНН), аванс, обеспечения, плюс
ИИ-оценки (`score`, `verdict`, `red_flags`) и разборы ТЗ.

Для аналитики **«где мы участвовали»** данных не было: факт участия, результат
(выиграли/проиграли), цена победителя и наша цена нигде не фиксировались.
Это ключевой пробел — без него аналитика участия невозможна.

## 2. Модуль аналитики

### Блок «Рынок» (работает сразу, чистый SQL)

- **Средний срок размещения** = `submission_deadline − publish_date`
  (средний/медианный; разрезы: регион, закон, способ, диапазон НМЦ, заказчик).
- **Стоимость**: распределение НМЦ, медиана по регионам/заказчикам, динамика
  помесячно, доля закупок с авансом и средний % аванса.
- **Профиль заказчиков**: кто чаще публикует профильные закупки, средние НМЦ и сроки.
- **Воронка релевантности**: загружено → relevant/maybe/noise → динамика по выгрузкам.

### Блок «Наше участие» (новые сущности)

Таблица `tender_participation`: статус (подали/не допущены/проиграли/выиграли),
наша цена, цена победителя, дата итогов, комментарий. Заполняется из веба.

- **Win rate** по сегментам.
- **Среднее снижение на торгах** (цена победителя / НМЦ) — главная цифра экономики.
- **Наша точность цены**: отличие нашей цены от победившей в проигранных торгах.
- **Корреляция ИИ-score с исходом** — валидация скоринга.

### Избранное

Таблица `tender_favorites` + звёздочка в списке/карточке + фильтр. В аналитике —
переключатель когорты: **вся база / relevant / избранные / участвовали**.

## 3. Fine-tuning vs RAG

Fine-tuning моделей Claude **недоступен** (Anthropic не предоставляет дообучение
весов через API) — и не подходит для задачи: мало примеров (сотни), экономика
устаревает поквартально, дообученная модель галлюцинирует цифры.

Правильная архитектура — **RAG + SQL-агрегаты**:

1. Для нового тендера SQL-ом отбираются похожие завершённые кейсы
   (регион + профиль + диапазон НМЦ; приоритет: участвовали → избранные → relevant).
2. SQL считает факты по сегменту: среднее снижение, win rate, типичный срок, аванс.
3. Claude получает карточку тендера + факты + 5–10 кейсов с исходами и заметками —
   и выдаёт структурированную рекомендацию: стратегия цены, ожидаемое снижение,
   риски, участвовать/нет. Модель рассуждает над реальными цифрами — предложения
   проверяемы.
4. Системный промпт (профиль компании + правила) кэшируется (prompt caching,
   ~10× дешевле на повторах). Массовые прогоны — Batch API (−50%).

Качество растёт по мере накопления `tender_participation` — без переобучения.

## 4. Пользователь в RAG-контуре (human-in-the-loop)

Единица RAG — **карточка-кейс**: тендер + исход участия + заметки людей + разбор ИИ.
Всё, что пользователь вводит в вебе, немедленно попадает в корпус (кейсы собираются
SQL-ом в момент запроса — переиндексация не нужна).

Пользователь вносит на карточке тендера:

1. **Участие** — структурированные факты (статус, цены, дата итогов).
2. **Заметки бюро** — свободный текст, попадает в промпт дословно
   («заказчик режет аванс», «ГИП оценил трудозатраты в 1.8х от сметы»).
3. **Оценка рекомендации** — «в точку / мимо» + комментарий; прошлые промахи
   включаются в промпт как контрпримеры (калибровка).

Цикл:

```
POST на /tender/{N} (участие/заметка)
  → Postgres (tender_participation / tender_notes / recommendation_feedback)
  → при запросе рекомендации: SQL-отбор похожих кейсов
  → промпт: [кэшируемый системный] + карточка + агрегаты + кейсы + фидбек
  → Claude → структурированная рекомендация → БД → карточка тендера
```

### Поиск похожих — сначала без векторов

На сотнях–тысячах тендеров хватает сегментного SQL-матчинга
(регион + профиль + НМЦ-диапазон) + `pg_trgm` по `subject`. Векторные эмбеддинги
(pgvector + внешний embeddings API, у Anthropic его нет) — отдельная фаза, когда
корпус вырастет.

## 5. Схема БД (миграции 0009–0010)

| Таблица | Поля |
|---|---|
| `tender_favorites` | reestr_number PK→tenders, note, created_at |
| `tender_participation` | reestr_number PK→tenders, status (applied/rejected/lost/won), our_price, winner_price, decided_at, comment, updated_at |
| `tender_notes` | id, reestr_number→tenders, text, created_at |
| `ai_recommendations` | id, reestr_number→tenders, model, recommendation JSONB, created_at |
| `recommendation_feedback` | id, recommendation_id→ai_recommendations, useful bool, comment, created_at |

## 6. Порядок работ

1. **Фаза A** — таблицы + миграции + кнопки в вебе (участие, заметки, избранное).
2. **Фаза B** — страница `/analytics`: рыночные метрики + метрики участия, когорты.
3. **Фаза C** — «ИИ-экономист»: RAG-рекомендация на карточке тендера + фидбек;
   фоновый прогон по паттерну doc_analysis_job.
